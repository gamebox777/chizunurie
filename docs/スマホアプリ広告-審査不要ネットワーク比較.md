# スマホアプリ広告 — 審査不要ネットワークの比較とアプリ化方針

chizunurie をスマホアプリ化（Android）した場合の広告ネットワーク選び。
特に「**事前審査なし／審査に通らなくても広告を入れられるか**」の観点で整理する。
最終更新：**2026-06-10**。

## 結論（一言）

- **事前審査なしで始められるのは Unity Ads（/ Unity LevelPlay・旧 ironSource）**。
  未公開アプリでも Game ID を発行すればすぐ実広告を配信できる。
- **AdMob・AppLovin・Meta Audience Network は審査／承認が必須**で、いずれも
  **ストア公開済みであることが前提**（未公開アプリは審査自体ができない）。
- ただし「広告ネットワークの審査」と「ストア（Google Play）の審査」は別物。
  Play に出すなら Play のアプリ審査は避けられない。完全に審査ゼロにできるのは
  Play 外配布（APK 直配布・審査の緩いストア）の場合のみ。

## 採用：Unity Ads（決定 2026-06-10）

事前審査なしで始められる Unity Ads を採用する。Unity Cloud ダッシュボード側の値
（**2026-06-10 ダッシュボードの「広告ユニット」画面で全 ID 確認済み**）：

| 項目 | Android | iOS |
|---|---|---|
| **Game ID** | `6133603` | `6133602` |
| Ad Unit：リワード動画 | `Rewarded_Android` | `Rewarded_iOS` |
| Ad Unit：バナー | `Banner_Android` | `Banner_iOS` |
| Ad Unit：インタースティシャル | `Interstitial_Android`（未使用） | `Interstitial_iOS`（未使用） |

- **動画広告はリワード型を使う**（インタースティシャルのユニットは存在するが使わない方針）。
- 現在実装済みなのは Android（`UnityAdsPlugin.java`）。iOS 版を作るときは上表の iOS 列の
  ID で同等のプラグインを実装する（`mobile/ios` は未生成）。

- Game ID は SDK 初期化（`UnityAds.initialize(context, "6133603", testMode, listener)`）に使う。
  開発中は `testMode=true` でテスト広告を出す（実機で実広告を出すとポリシー違反になり得る）。
- 報酬付与のサーバー検証は **S2S リワードコールバック**（ダッシュボードで URL 設定・
  `sid`＝userId と署名 `hmac` 付きで backend を叩く）に置き換え可能。Web 版の
  nonce 方式（`issueVideoRewardNonce`/`claimVideoReward`）からの移行先。

### 実装状況（2026-06-10 実装・エミュレータ検証済み）

コードは一式実装済み。詳細は [mobile/README.md](../mobile/README.md) の「Unity Ads」節を参照。

- **ネイティブ**：`mobile/android/app/src/main/java/jp/chizunurie/app/UnityAdsPlugin.java`
  （Capacitor プラグイン `UnityAds`・`showRewarded()`）。`MainActivity` で登録。
  `app/build.gradle` に `com.unity3d.ads:unity-ads:4.18.1` と
  `buildFeatures { buildConfig true }`（`BuildConfig.DEBUG`＝テストモード判定に必要）。
- **フロント**：`frontend/src/lib/nativeRewardedAd.ts`（新規）が
  `window.Capacitor.Plugins.UnityAds.showRewarded()` を呼び、Web 版 GPT と同じ
  `{ outcome, detail? }` を返す。`Map.tsx` の `openVideoReward` が `isNativeApp()` で
  Unity Ads / GPT を出し分け。**「▶ 動画を見て回復」ボタンはアプリ内のみ表示**
  （Web は AdSense 審査通過まで非表示のまま）。報酬付与は Web 版と同じ nonce 方式。
- **フッターバナー（2026-06-10 追加・エミュレータ検証済み）**：`UnityAdsPlugin` に
  `showBanner()`/`hideBanner()` を追加。320x50 の `BannerView` を画面下中央にネイティブ表示し、
  **WebView を bottomMargin で持ち上げて場所を確保**する（Web 側の CSS 調整は不要）。
  フロントは `frontend/src/lib/nativeBannerAd.ts`（新規）＋ `Map.tsx` がアプリ内のみ
  マウント時に表示（失敗時は1分おき・最大5回再試行）。あわせて、アプリ内では地図トップの
  サイトフッター（`SiteFooter` variant="bar"・クローラ導線）を非表示にした（Web はそのまま）。
- **mobile/android を git 管理に変更**：手書きのネイティブコードを含むため
  ルート `.gitignore` の `mobile/android/` 除外を解除（生成物は `android/.gitignore` が除外）。

検証（Android エミュレータ）：

- **debug ビルドは Unity 公式テスト用ゲーム（Game ID `14851`・placement
  `rewardedVideo`/`bannerads`）に固定**してあり、**テスト広告で全機能が動く状態**：
  - リワード：「▶ 動画を見て回復」→ nonce 発行 → テスト広告視聴 → granted →
    backend 付与 → 残高反映（35/35 → 70/35）・クールダウン表示まで一連を実機確認済み。
  - バナー：起動時に自動表示・WebView が持ち上がる・`hideBanner()` で復帰。
  - アプリ内のサイトフッター非表示も確認済み。
- **本番 Game ID `6133603` は、広告ユニット（Rewarded_Android 等）作成後も load が
  `Network error occurred`（PUBLIC_ERROR_CODE_INIT_UNKNOWN）で失敗する**（2026-06-10 夜時点）。
  初期化（`Initialized successfully`）までは通る。エミュから Unity の全エンドポイント
  （config/auction/gateway/cdn）への疎通は確認済み。ユニット作成直後の反映待ちの可能性あり →
  **時間を置いて再テストする**。release ビルド（=本番 ID・テストモード OFF）はこれが直るまで
  広告が出ない（バナーは出ず・リワードは「動画を出せない」トースト。クラッシュはしない）。

### ビルド（version 1.2 / versionCode 3・2026-06-10）

- debug APK：`mobile/android/app/build/outputs/apk/debug/app-debug.apk`（テストゲームでフル動作）
- 署名済み release AAB：`mobile/android/app/build/outputs/bundle/release/app-release.aab`
  （`npm run bundle`・Play 提出用。ただし広告は上記の本番 ID 問題が直るまで出ない）

## 各ネットワークの審査要件（2026年時点）

| ネットワーク | 事前審査 | ストア公開の要否 | 備考 |
|---|---|---|---|
| **Unity Ads / LevelPlay** | **なし** ✅ | 不要（未公開でも配信可） | ゲーム特化でリワード動画の在庫が豊富。eCPM は AdMob よりやや低め傾向 |
| **AdMob**（Google） | あり（app readiness review・通常2〜3日） | **必須**（対応ストアに掲載＋アカウントにリンク） | 通過まで limited ad serving。2025年1月以降 **app-ads.txt 検証も必須** |
| **AppLovin MAX** | あり（アカウント承認・厳しめ） | **必須**（Play 記載のメールからの確認も必要） | 未公開アプリは承認不可と明言 |
| **Meta Audience Network** | あり | 必須 | 「審査なし」枠には入らない |

### AdMob の補足

- AdMob のアプリ審査は **AdSense「サイト」審査（2026-06-10 にコンテンツなしで否認されたもの）
  とは別系統**。中身がちゃんとしたゲームアプリなら通常は普通に通る。
  「審査が厳しい」というより「**ストア公開が先**」という順序の制約。
- app-ads.txt は Web 版 ads.txt（`gamebox777.org`）の運用経験がそのまま活きる
  （開発者サイトのルートに `app-ads.txt` を置き、ストア掲載の開発者 URL と一致させる）。

## chizunurie に当てはめたときの注意

1. **TWA では中身が Web ページなのでネイティブ広告 SDK は使えない**。
   TWA は Chrome がそのまま Web を表示する方式のため、AdMob SDK / Unity Ads SDK を
   組み込めない（広告は Web 版 GPT リワードのまま＝AdSense 審査問題から逃げられない）。
   アプリ内に SDK 広告を入れるなら **Capacitor（WebView ＋ ネイティブプラグイン）構成**が必要。
   Capacitor には AdMob・Unity Ads ともコミュニティプラグインがある。
2. **報酬付与の検証**：AdMob のリワードには **SSV（サーバーサイド検証ポストバック）がある**
   （Web GPT には無かったもの）。Unity Ads にも S2S コールバックがある。アプリ化すれば
   Web 版の自前 nonce 方式より強い不正対策に置き換えられる（既存の
   `issueVideoRewardNonce`/`claimVideoReward` は流用可能）。
3. Google Play の Families ポリシー等、**ストア側の広告ポリシー**は別途かかる
   （対象年齢設定・広告の出し方の申告など）。

## 現実的な進め方（2案）

- **案A：すぐ審査なしで試す** — Capacitor 化 → **Unity Ads** のリワード動画。
  ストア公開前から実広告でテストでき、公開後もそのまま使える。
- **案B：単価重視** — まず Play にアプリ公開 → **AdMob** の readiness 審査（2〜3日）
  ＋ app-ads.txt 設置。

両方やる（AdMob メディエーションや LevelPlay で Unity Ads を束ねる）のも一般的。
まず案Aで配信実績を作り、Play 公開後に案Bを足してメディエーション化が無理のない順序。
