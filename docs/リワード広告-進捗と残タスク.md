## ⚠️ 2026-06-10：サイト審査 否認 —「パブリッシャーのコンテンツを含まない画面」

AdSense「サイト」の審査が **ポリシー違反（要確認）** で否認された。理由は
「コンテンツがない／有用性の低いコンテンツ／作成中／行動（ナビゲーション等）が目的の画面」。

### 原因の見立て

- **サブドメインだから、ではない**。登録サイトはルート `gamebox777.org` で、審査クローラは
  apex を見にいく。apex は `chizunurie.gamebox777.org` へリダイレクトされ、その先は
  **地図キャンバス＋ボタンだけの SPA**。クロールして得られるテキストがほぼ無く、
  「ゲーム/ツール画面単体のページ」としてコンテンツなし判定された。
- 当時のフロントの公開ページは地図トップ・admin・delete-account のみで、
  紹介・遊び方・プライバシーポリシー等の「読めるページ」がゼロだった。
- 別ドメインに移してルート直下にアプリを置いても、クローラが見るものは同じなので
  **ドメイン構成の変更だけでは解決しない**。

### 対応（2026-06-10 実装済み・フロントのコード）

クローラが読める SSR テキストページ一式と導線を追加した：

| ページ | パス | ファイル |
|---|---|---|
| ゲーム紹介 | `/about` | `frontend/src/app/about/page.tsx` |
| 遊び方 | `/how-to-play` | `frontend/src/app/how-to-play/page.tsx` |
| プライバシーポリシー | `/privacy` | `frontend/src/app/privacy/page.tsx`（広告 Cookie・GA・位置情報・opt-out を明記） |
| 運営者情報・問い合わせ | `/contact` | `frontend/src/app/contact/page.tsx` |
| 更新情報 | `/news` | `frontend/src/app/news/page.tsx` |
| コラム（索引＋2本） | `/columns`, `/columns/mesh-1km`, `/columns/walking-tips` | `frontend/src/app/columns/**` |

- **地図トップ最下部に極薄フッター**（`SiteFooter` variant="bar"）で全ページへリンク
  （クローラ導線）。各情報ページ末尾にも相互リンクフッター。
- `sitemap.xml` / `robots.txt`（`app/sitemap.ts` / `app/robots.ts`・`/admin` は disallow）。
- `delete-account` のプライバシーポリシーリンクを外部（unitygamebox.com）→ 自サイト `/privacy` へ変更。
- フッターラベルの i18n（ja/en）を `lib/i18n.tsx` に追加。

### 再審査の手順（デプロイ後）

1. 本番デプロイし、`https://chizunurie.gamebox777.org/about` 等が SSR テキストで返ることを確認。
2. AdSense「サイト」→ gamebox777.org →「**問題を修正しました**」にチェック →「**審査をリクエスト**」。
3. 審査には数日〜2週間かかることがある。

### それでも否認が続く場合の次の一手

- **apex のリダイレクトをやめて、apex に実体のあるポータルページを置く**
  （ゲーム紹介＋各情報ページへのリンク。静的1ページで可・Cloudflare Pages/Worker でも可）。
  審査クローラの入口である apex 自体にコンテンツが載る。ads.txt Worker 問題も自然に解消する。
- 専用ドメイン（chizunurie.com 等）への移行は最後の手段（GAM サイト設定・ads.txt・OAuth
  リダイレクト URL・Coolify の設定やり直しが必要）。

## いちばん重要な技術的結論：Web の GPT リワードに SSV は無い

- **アプリ（AdMob）**のリワードには、Google がサーバーへ署名付きで叩く **SSV ポストバック**がある。
- **Web の GPT リワードは報酬付与がクライアントイベント `rewardedSlotGranted` で完結**する（公式仕様）。
- したがって GAM 広告ユニットの **「サーバーサイドで認証されるポストバック URL」はメディエーション
  （アプリ SDK 経由）専用**で、今回の Web 配信では**発火しない**。→ **空欄のままで正しい。**
- 不正対策は **自前 nonce ＋ サーバー側のクールダウン/日次上限**で実装した（下記）。

### ドメインの粒度（apex vs サブドメイン）— Worker 方式に変更

AdSense の登録サイトは **ルート `gamebox777.org`**。Google は `https://gamebox777.org/ads.txt`
（ルート直下）を見にいくため、apex 側でも ads.txt に到達させる必要がある。

**当初の 301 リダイレクト方式は所有権確認を通らなかった**：apex を Cloudflare の Redirect Rule で
サブドメイン `chizunurie.gamebox777.org/ads.txt` へ 301 していた。curl では Google のクローラ UA
（`Google-Adstxt`/`AdsBot-Google`/`Mediapartners-Google`）でも 301→200 が正しく返り、Cloudflare の
ボット対策にも弾かれていない。にもかかわらず AdSense の「確認」は **「お客様のサイトは確認できませんでした」**
で失敗した。販売者クロールは同一ルートドメイン内リダイレクトを追従するが、**所有権確認の経路は apex 自身に
実体（200）を求める**（＝apex→別ホストの 301 を通さない）ためと判断。

→ **apex `gamebox777.org/ads.txt` を Cloudflare Worker で「実体200」として直接返す方式に変更**。
リダイレクト追従・クロールキャッシュの問題を両方つぶす。Worker コードは
[infra/cloudflare/apex-ads-txt.worker.js](../infra/cloudflare/apex-ads-txt.worker.js)。

Cloudflare 側の作業（`unitygamebox.com` には未着手）：

- apex の `AAAA 100::`（Proxied）レコードはそのまま残す（apex を Cloudflare 経由にするため）。
- **既存の Redirect Rule「apex ads.txt redirect」を無効化／削除**する。
  ※ Redirect Rules は Workers より前に評価されるため、残っていると 301 が先に発火して Worker が呼ばれない。
- Worker を作成し、Route `gamebox777.org/ads.txt`（Zone: `gamebox777.org`）を割り当てる。

期待する確認結果（変更後）：
```
$ curl -sI https://gamebox777.org/ads.txt
HTTP/2 200  (content-type: text/plain, content-length: 59)   ← 301 が消えて直接 200
```

## 現実的な注意

- Web のリワードはフィル率・単価がアプリより低め。未充填時はトースト案内で逃がす実装にしてある
  （`unavailable` / `dismissed`）。
- 高負荷の不正対策は Web では限界があるが、**nonce（単回使用）＋ 30分クールダウン＋1日5回上限**で
  乱用は実用上の範囲に抑えている。
