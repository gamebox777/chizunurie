# デプロイ手順（Coolify 全部寄せ・単一ドメイン構成）

> **Coolify を使うなら通常は [Coolifyデプロイ-compose.md](Coolifyデプロイ-compose.md) が簡単。**
> Coolify に `docker-compose.coolify.yml` を1つ指定するだけで、フロント・バック・DB を
> まとめてデプロイでき、ドメイン・TLS・DBパスワード・認証シークレットも自動。
> このページ（下記）は、サービスを **3つ個別のリソースとして** 細かく運用したい場合の手順。
>
> Coolify を使わずローカルや単一VPSで `docker compose` 一発なら → [Docker起動.md](Docker起動.md)。
>
> いずれの方法でも **DBマイグレーションはバックエンド起動時に自動適用される**
> （`backend/docker-entrypoint.sh`）。下の「DBマイグレーション」を手動で流す必要は基本ない。

VPS 上の Coolify に **フロント(Next.js)・バックエンド(Hono)・PostgreSQL** の3つを置き、
**1つの公開ドメイン**で配信する構成。ブラウザはフロントの同一オリジンだけを叩き、
`/api/*` は Next.js の rewrites がサーバー側でバックエンドへ中継する（CORS/Cookie 問題が起きない）。

```
                 https://chizunurie.unitygamebox.com
ブラウザ ──────────────▶ フロント(Next.js / Coolify)
                              │  rewrites() でサーバー側中継
                              │   /api/auth/*     → backend /api/auth/*
                              │   /api/backend/*  → backend /*
                              ▼
                         バックエンド(Hono / Coolify, :3001)
                              │  Docker 内部ネットワーク
                              ▼
                         PostgreSQL(Coolify, 外部非公開)
```

前提：独自ドメインあり / Google OAuth は開発・本番で同じ Client ID/Secret を使う。

---

## 0. 事前準備（Google OAuth）

Google Cloud Console → 該当 OAuth クライアントの「承認済みのリダイレクト URI」に
**本番URLを追加**する（開発用の localhost はそのまま残す）：

```
http://localhost:3001/api/auth/callback/google        ← 既存（開発用・残す）
https://chizunurie.unitygamebox.com/api/auth/callback/google ← 追加（本番用）
```

> ドメインは自分のものに置き換える。以降の `chizunurie.unitygamebox.com` も同様。

---

## 1. PostgreSQL を作成

1. Coolify → プロジェクト作成（例：`chizunurie`）
2. 「+ New Resource」→ **PostgreSQL** を追加
3. 名前を控える（内部ホスト名になる。例：`chizunurie-postgres`）／DB名・ユーザー・パスワードを設定
4. **外部公開はしない**（Public Port を開けない）。同一プロジェクト内のサービスから
   内部ホスト名で接続できる
5. 接続文字列を控える：
   ```
   postgres://<user>:<password>@chizunurie-postgres:5432/<db>
   ```

---

## 2. バックエンド(Hono)をデプロイ

1. 「+ New Resource」→ **Application** → Git リポジトリ連携
2. **Build Pack: Dockerfile** を選択、**Base Directory を `/backend`** に設定
   （[backend/Dockerfile](../backend/Dockerfile) を使う）
3. 名前を控える（フロントの中継先になる。例：`chizunurie-backend`）
4. ポート：**3001**（Exposed Port）。ドメインは付けなくてよい（内部利用のみ）
5. Environment Variables（[backend/.env.production.example](../backend/.env.production.example) 参照）：
   | 変数 | 値 |
   |---|---|
   | `DATABASE_URL` | `postgres://<user>:<pw>@chizunurie-postgres:5432/<db>` |
   | `BETTER_AUTH_SECRET` | `openssl rand -base64 32` で新規生成（本番用） |
   | `BETTER_AUTH_URL` | `https://chizunurie.unitygamebox.com`（公開ドメイン） |
   | `FRONTEND_URL` | `https://chizunurie.unitygamebox.com`（同上） |
   | `GOOGLE_CLIENT_ID` | 開発と同じ値 |
   | `GOOGLE_CLIENT_SECRET` | 開発と同じ値 |
   | `PORT` | `3001` |
   | `NODE_ENV` | `production`（Secure Cookie 有効化に必要） |
6. デプロイ

### DB マイグレーション

**通常は不要。** バックエンドのコンテナは起動時に `docker-entrypoint.sh` が
`node dist/migrate.js`（`backend/drizzle/*.sql` を適用）→ サーバー起動 の順で実行するため、
空のDBでも初回デプロイでテーブルが自動で揃う。スキーマを変えたときは
`cd backend && npm run db:generate` で新SQLを生成・コミットすれば、次回デプロイで自動適用される。

手動で流したい場合（自動適用を切る・差分確認したい等）はどちらか：

- ローカルから本番DBに対して push（DBを一時的に外部公開 or SSHトンネル経由）
  ```bash
  cd backend
  DATABASE_URL="postgres://...@<本番DB>:5432/<db>" npm run db:push
  ```
- 生成済みSQLを `npm run db:migrate` で適用

---

## 3. フロント(Next.js)をデプロイ

1. 「+ New Resource」→ **Application** → 同じ Git リポジトリ
2. **Build Pack: Dockerfile**、**Base Directory を `/frontend`** に設定
   （[frontend/Dockerfile](../frontend/Dockerfile) を使う）
3. ポート：**3000**
4. **Domains** に公開ドメインを設定：`https://chizunurie.unitygamebox.com`
   （Coolify が Let's Encrypt で TLS を自動取得）
5. **Build Variable（ビルド時引数）** に以下を設定 ★重要★
   | 変数 | 値 | 種別 |
   |---|---|---|
   | `BACKEND_URL` | `http://chizunurie-backend:3001` | **Build Variable** |
   | `NEXT_PUBLIC_APP_ENV` | `production`（省略可・Dockerfile 既定値） | **Build Variable** |

   > `BACKEND_URL` は Next.js の `rewrites()` がビルド時に焼き込むため、
   > 実行時の環境変数ではなく **Build Variable（Docker の ARG）** として渡す必要がある。
   > 値はバックエンドサービスの内部ホスト名（手順2で控えた名前）。
   >
   > `NEXT_PUBLIC_APP_ENV` はデバッグメニューの実行モード表示用。Dockerfile の
   > 既定が `production` なので**本番では設定不要**。値が `production` だとモード表示が
   > 「本番」になる（フル Docker 版は compose 側で `docker` を渡して「フル Docker」と表示）。
6. デプロイ

---

## 4. 動作確認

1. `https://chizunurie.unitygamebox.com/` が表示される
2. ヘルスチェック（プロキシ経由）：ブラウザで地図が出て塗りが保存できる
3. ログイン：メール/パスワード、Google OAuth の両方を確認
   - Google ログイン後に `https://chizunurie.unitygamebox.com` に戻り、セッションが保持されること
4. リロードしてもログイン状態が維持される（Cookie が効いている）こと

---

## トラブルシューティング

- **Google ログイン後に redirect_uri_mismatch**：手順0の本番リダイレクトURIが
  Google 側に登録されているか、`BETTER_AUTH_URL` が公開ドメインと一致しているか確認。
- **ログインできるがリロードでセッションが消える**：`NODE_ENV=production` が
  バックエンドに設定されているか（Secure Cookie）、`BETTER_AUTH_URL`/`FRONTEND_URL` が
  https の公開ドメインになっているか確認。
- **フロントから API に繋がらない（502/接続拒否）**：`BACKEND_URL` を **Build Variable** で
  設定したか（実行時 env だけでは効かない）。設定後は **再ビルド** が必要。
  値がバックエンドの内部ホスト名・ポート(3001)と一致しているか確認。
- **地図データ(PMTiles)が出ない**：`japan.pmtiles` は git 管理されており `public/` に含まれる。
  リポジトリに存在するか（`git ls-files frontend/public/data/japan.pmtiles`）確認。
- **DB に繋がらない**：`DATABASE_URL` の内部ホスト名が Coolify の Postgres リソース名と
  一致しているか、同一プロジェクト内に配置されているか確認。
